Arquitectura POC

Agente de compra para componentes electronicos. Busca precio y tiempo de envio del producto que el usuario quiere comprar, primero en nuestro catalogo y, si no alcanza, en proveedores externos mediante web search.

Objetivo del sistema

Entrada

Consulta tecnica

El usuario escribe en el chat que producto electronico quiere comprar o comparar.

Proceso

Orquestacion inteligente

El planificador clasifica la consulta, evita trabajo innecesario y usa web solo si la busqueda local no alcanza.

Salida

Recomendacion accionable

Devuelve producto encontrado, precio, stock/disponibilidad, tiempo estimado de envio, proveedor y nivel de confianza.

Estado actual del codigo

Implementado - Catalogo local con RAG

CatalogAgent consulta CatalogRetriever, que usa ChromaDB persistente sobre el catalogo de componentes.

Implementado - Web search externo

WebResearchAgent usa WebSearchAdapter. En la arquitectura objetivo sera fallback para buscar proveedores externos con SerpAPI/Google Shopping.

Implementado - Seguridad basica

validate_user_query bloquea entradas vacias, largas, secretos, payloads peligrosos y prompt injection basica.

Implementado - UI, metricas y auditoria

app/main.py expone Streamlit. Metrics mide latencias, contadores y scores. Cada respuesta se registra en logs/advisor_audit.jsonl.

Falta - LLM real para agentes

Hoy la respuesta final se arma con reglas. La capa objetivo sera LangChain + Groq para clasificar intencion, decidir el camino de busqueda y verificar la respuesta.

Contrato entre agentes

Cada buscador debe devolver JSON simple: product_name, sku, price, shipping_eta, vendor, evidence, source, confidence y failure_mode.

Por que esta arquitectura

Velocidad

Evitar llamadas innecesarias

El planner primero clasifica la consulta. Si no hace falta buscar, responde rapido sin RAG ni web search.

Experiencia

Usuario informado

Mientras hay busquedas, el sistema muestra mensajes streaming para que el usuario entienda que esta pasando.

Costo

Web como fallback

Google Shopping/SerpAPI se usa solo si el catalogo local no resuelve el producto, precio o envio.

Flujo de datos

1. Consulta Texto del usuario: producto a comprar, SKU, categoria, precio, envio o pregunta sobre el agente.
2. Validacion Sanitizacion, longitud maxima, bloqueo de secretos y ataques obvios.
3. Orquestador Clasifica, emite streaming, ejecuta busqueda local hibrida y decide si usar fallback web.
4. Evidencia Candidatos de RAG + keyword/BM25. Si fallan, candidatos externos de Google Shopping.
5. Observabilidad Langfuse recibe spans de planner, RAG, web, verifier, tokens, costo y errores.
6. Respuesta y auditoria Producto, precio, envio, proveedor, fuente, limitaciones y evento JSONL.

Observabilidad con Langfuse

Tracing

Una traza por consulta

Cada request crea una traza con spans para validacion, planner, RAG, keyword/BM25, fallback web, agregacion, verificacion y auditoria.

Costos

Tokens y latencia

Se registran tokens de entrada/salida, modelo Groq usado, costo estimado, latencia por modulo y latencia end-to-end.

Calidad

Scores evaluables

Se envian scores como Recall@k, Context Recall y abstencion correcta.

Metricas necesarias

Se usa un set reducido de metricas para que la evaluacion sea simple, explicable y defendible en el informe.

Grupo Metrica Como medirla en este proyecto Meta inicial
Retrieval Recall@k Conjunto eval: para cada producto, verificar si el SKU/producto correcto aparece en top-k de la busqueda local hibrida. >= 0.85 @ top-3
Generacion Context Recall Verificar si la respuesta usa la evidencia necesaria: producto, precio, envio, proveedor y fuente. >= 0.85
Generacion Abstencion correcta Preguntas fuera de dominio o sin evidencia suficiente: el sistema debe responder rapido o abstenerse. >= 0.90
Robustez Caida ante parafrasis / ruido Comparar Recall@k, Context Recall y abstencion entre pregunta limpia, parafrasis, typos y consulta larga. Delta <= 10%
Operacion Latencia end-to-end p50/p95 Medir desde entrada hasta respuesta final. Separar latencia RAG, web y LLM. p50 < 5s, p95 < 12s
Operacion Costo por respuesta Tokens de Groq + llamadas SerpAPI. Web search debe quedar visible como costo de fallback. Visible por request

Brechas contra la consigna

Cubierto

  • Interfaz funcional con Streamlit.
  • Catalogo local con RAG y busqueda web externa.
  • RAG con ChromaDB persistente.
  • Seguridad de entrada basica.
  • Modelo preexistente para intencion con fallback.
  • Auditoria automatica en JSONL.

Pendiente

  • LLM real para planificacion, agregacion y verificacion.
  • Clasificacion rapida de rutas: fuera de dominio, ayuda del agente, local hibrido o fallback web.
  • Keyword/BM25 para complementar RAG antes de usar web search.
  • Metricas de calidad RAG/generacion con dataset de evaluacion.
  • Politicas de output: precio, envio, proveedor, citas, abstencion y formato JSON validable.
  • Optimizacion de costo: web search solo como fallback, cache, limites de tokens y seleccion dinamica.

Diseno tecnico de implementacion

Framework

LangGraph + LangChain

LangGraph orquesta rutas, estado compartido, nodos condicionales y fallback. LangChain aporta prompts, modelos Groq, structured output y callbacks.

LangGraph LangChain Groq Langfuse Streamlit
No A2A

Agentes como nodos

No usamos A2A porque todos los agentes viven en el mismo backend, comparten estado y necesitan baja latencia. La separacion sera por nodos y contratos JSON.

Runtime

Python local, sin Docker

La app corre con Python y Streamlit. Langfuse se usa mediante SDK Python contra la version web/cloud configurada en .env.

Prompts

Markdown en ingles

Todos los prompts LLM viven en llm-agent-project/prompts, estan escritos en ingles y se versionan junto al repo. No se hardcodean prompts en Python.

ChatInput -> PlannerNode # LLM: clasifica ruta y normaliza consulta -> RouteDecision -> DirectResponseNode # fuera de dominio / ayuda del agente -> LocalHybridSearchNode # RAG + Keyword/BM25 en paralelo, sin LLM -> LocalEvidenceGate # regla deterministica -> AggregatorVerifierNode # LLM: respuesta final si hay evidencia local -> WebFallbackNode # SerpAPI / Google Shopping, sin LLM -> AggregatorVerifierNode -> FinalResponse -> LangfuseTrace + AuditLog

Escenarios y llamadas LLM

Escenario Ruta Llamadas LLM Motivo
Fuera de dominio PlannerNode -> DirectResponseNode 1 El planner responde rapido que el agente solo busca productos electronicos para compra.
Pregunta sobre el agente PlannerNode -> DirectResponseNode 1 El planner responde desde instrucciones del system prompt. No usa RAG ni web.
Producto encontrado localmente PlannerNode -> LocalHybridSearchNode -> AggregatorVerifierNode 2 La busqueda local hibrida encuentra producto, precio/envio o suficiente evidencia.
Producto no encontrado localmente PlannerNode -> LocalHybridSearchNode -> WebFallbackNode -> AggregatorVerifierNode 2 Web search no requiere LLM. El verificador usa resultados locales y externos para responder.

Modelos configurables

Streamlit tendra un dropdown por agente. La lista sale de LLM_MODEL_OPTIONS, separada por comas, para poder cambiar modelos sin tocar codigo.

Variable Uso Default sugerido
LLM_MODEL_OPTIONS Lista de modelos disponibles en los dropdowns. groq/compound-mini,groq/compound,openai/gpt-oss-20b,openai/gpt-oss-120b
PLANNER_MODEL Modelo por defecto del planner/router. groq/compound-mini
VERIFIER_MODEL Modelo por defecto de agregacion + verificacion. groq/compound
DIRECT_RESPONSE_MODEL Modelo para respuestas rapidas fuera de dominio o sobre el agente. groq/compound-mini

Condicion de fallback web

El fallback externo se decide con reglas, no con otra llamada LLM. Esto reduce costo y latencia.

use_web_fallback = WEB_FALLBACK_ENABLED and ( local_results_count < LOCAL_SEARCH_MIN_RESULTS or best_local_score < LOCAL_SEARCH_MIN_SCORE or missing(price) or missing(shipping_eta) )

Como replicar el POC

Configuracion

  1. Crear llm-agent-project/.env desde .env.example.
  2. Agregar GROQ_API_KEY para el LLM.
  3. Agregar LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY y LANGFUSE_HOST para observabilidad.
  4. Agregar claves web si se usara busqueda real: TAVILY_API_KEY, SERPAPI_API_KEY, ZYTE_API_KEY.

Ejecucion

  1. cd llm-agent-project
  2. pip install -r requirements.txt
  3. python scripts/ingest_catalog_to_chroma.py
  4. streamlit run app/main.py
