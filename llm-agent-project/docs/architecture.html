Agente para busqueda de precios de componentes electronicos

Agente de compra para componentes electronicos. Busca precio y URL de compra del producto que el usuario quiere comprar, primero en nuestro catalogo y, si no alcanza, en proveedores externos mediante web search.

Objetivo del sistema

Entrada

Consulta tecnica

El usuario escribe en el chat que producto electronico quiere comprar o comparar.

Proceso

Orquestacion inteligente

El planificador clasifica la consulta, evita trabajo innecesario y usa web solo si la busqueda local no alcanza.

Salida

Recomendacion accionable

Devuelve producto encontrado, precio, URL de compra, proveedor/fuente y nivel de confianza.

Estado actual del codigo

Implementado - Busqueda local hibrida

LocalHybridSearchAgent combina RAG semantico con ChromaDB y busqueda Keyword/BM25. El merge favorece resultados que aparecen por ambas senales.

Implementado - Web search externo

WebResearchAgent usa WebSearchAdapter. Se activa como fallback con SerpAPI/Google Shopping cuando la evidencia local no alcanza.

Implementado - Seguridad basica

validate_user_query bloquea entradas vacias, largas, secretos, payloads peligrosos y prompt injection basica.

Implementado - Chat, metricas y auditoria

app/main.py expone Streamlit como chat con historial. Cada respuesta muestra tokens, costo estimado, tiempos por etapa, resultados por tool y auditoria JSONL.

Implementado - LLM real con LangGraph + LangChain + Groq

El planner y el agregador/verificador usan Groq mediante LangChain. Para rutas fuera de scope, el planner devuelve tambien la respuesta final en direct_answer, evitando una segunda llamada LLM.

Contrato entre agentes

Los nodos intercambian estado estructurado: decision del planner, resultados RAG, resultados Keyword/BM25, decision del evidence gate, resultados web, respuesta verificada y metricas.

Por que esta arquitectura

Velocidad

Evitar llamadas innecesarias

El planner primero clasifica la consulta. Si no hace falta buscar, responde rapido sin RAG ni web search.

Experiencia

Usuario informado

Mientras hay busquedas, el sistema muestra progreso por etapa para que el usuario entienda que esta pasando. El streaming actual es de estado, no de tokens del LLM.

Costo

Web como fallback

Google Shopping/SerpAPI se usa solo si el catalogo local no resuelve el producto, precio o URL de compra.

Tokens

Cortes tempranos

Las rutas condicionales evitan pasos caros: out of scope saltea RAG/web/verificacion, y el evidence gate decide fallback web sin otra llamada LLM.

Planificacion de ejecucion

Consulta en chat El usuario envia una pregunta en Streamlit. La UI guarda historial en st.session_state["messages"] y manda los ultimos mensajes al backend para resolver referencias del turno actual.
Clasificacion rapida El planner decide si la consulta esta fuera de dominio, es sobre el agente/compania o requiere busqueda de compra. Tambien produce normalized_query, product_terms y web_search_query en ingles.
Rutas sin busqueda Si esta fuera de dominio, responde que el agente solo ayuda a buscar productos electronicos para comprar. Si es sobre el uso del agente, responde desde el system prompt.
Busqueda local hibrida Para productos, corre RAG semantico y keyword/BM25 sobre nuestro catalogo, luego fusiona candidatos en hybrid_merge. Esta es la ruta preferida.
Llamando RAG semantico sobre el catalogo local...
Haciendo keyword/BM25 search sobre SKU, nombre y descripcion...
Evidence gate y fallback web El evidence gate decide con reglas deterministicas. Si falla la evidencia local, se consulta Google Shopping con SerpAPI usando el query web en ingles.
Evaluando si la evidencia local alcanza o si hace falta fallback web...
Local evidence is insufficient. Searching external suppliers with Google Shopping.
Agregacion + verificacion El verificador compara candidatos, valida fuente, precio, URL de compra y confianza. La UI muestra un expander por respuesta con metricas y resultados por tool.

Observabilidad con Langfuse

Tracing

Una traza por consulta

Cada request crea una traza con spans para validacion, planner, RAG, keyword/BM25, fallback web, agregacion, verificacion y auditoria.

Costos

Tokens y latencia

Se registran tokens de entrada/salida, modelo Groq usado, costo estimado, latencia por modulo y latencia end-to-end.

Calidad

Metricas evaluables

La app registra evidencia para evaluar Recall@k, Context Recall y abstencion correcta. El calculo offline queda como paso de informe.

Evento Langfuse Contenido Uso en el informe
trace Consulta, respuesta final, usuario/sesion anonima, metadata de entorno. Demostrar flujo completo por request.
span Planner, RAG, keyword/BM25, WebResearchAgent, Verifier, auditoria. Identificar cuellos de botella y fallos por componente.

Diseno tecnico de implementacion

Framework

LangGraph + LangChain

LangGraph orquesta rutas, estado compartido, nodos condicionales y fallback. LangChain aporta prompts, modelos Groq, structured output y callbacks.

LangGraph LangChain Groq Langfuse Streamlit
Runtime

Python local, sin Docker

La app corre con Python y Streamlit. Langfuse se usa mediante SDK Python contra la version web/cloud configurada en .env.

Inferencia preentrenada

Clasificador auxiliar

PretrainedIntentModel encapsula facebook/bart-large-mnli como modelo preexistente para intencion tecnica. Actualmente esta disponible como modulo, pero debe conectarse al grafo para quedar demostrado en el flujo ejecutable.

Tool final

Telegram controlado por LangGraph

send_telegram_message esta registrado con @tool de LangChain, pero no queda a decision libre del LLM. El nodo telegram_notification lo ejecuta solo si la UI lo habilita y despues del verificador hay producto, precio y URL de compra.

ChatInput + ChatHistory -> PlannerNode # LLM: clasifica ruta, normaliza consulta y prepara web_search_query en ingles -> RouteDecision -> END # fuera de dominio / ayuda del agente: direct_answer sale del planner -> LocalHybridSearchNode # RAG + Keyword/BM25 en la etapa local, sin LLM -> rag_search -> keyword_bm25_search -> hybrid_merge -> LocalEvidenceGate # regla deterministica -> AggregatorVerifierNode # LLM: respuesta final si hay evidencia local -> WebFallbackNode # SerpAPI / Google Shopping, sin LLM -> AggregatorVerifierNode -> TelegramNotificationNode # LangChain tool: send_telegram_message, controlado por reglas -> FinalResponse + StreamlitDetails -> LangfuseTrace + AuditLog + Telegram + TimingBar

Optimizacion implementada

Las rutas out_of_domain y agent_info terminan en el planner: el mismo JSON incluye direct_answer. Asi se elimina DirectResponseNode para esos casos y se baja de 2 a 1 llamada LLM.

Escenarios y llamadas LLM

Escenario Ruta Llamadas LLM Motivo
Fuera de dominio PlannerNode -> END 1 El planner clasifica, devuelve direct_answer y se saltean RAG, BM25, web, verificador, Telegram y una segunda llamada LLM.
Pregunta sobre el agente PlannerNode -> END 1 El planner detecta que es informacion del agente, devuelve direct_answer y evita busqueda y segunda llamada LLM.
Producto encontrado localmente PlannerNode -> LocalHybridSearchNode -> AggregatorVerifierNode 2 La busqueda local hibrida encuentra producto, precio y URL de compra local. El evidence gate evita web search y una llamada externa.
Producto no encontrado localmente PlannerNode -> LocalHybridSearchNode -> WebFallbackNode -> AggregatorVerifierNode 2 Web search no requiere LLM. El verificador usa resultados locales y externos para responder; el fallback se activa solo si la evidencia local no alcanza.

Condicion de fallback web

El fallback externo se decide con reglas, no con otra llamada LLM. Esto reduce costo y latencia.

use_web_fallback = WEB_FALLBACK_ENABLED and ( local_results_count < LOCAL_SEARCH_MIN_RESULTS or best_local_score < LOCAL_SEARCH_MIN_SCORE or specific_product_terms_missing_from_best_result or missing(price) or missing(price) )

Terminos especificos son tokens como SKUs, codigos con numeros o identificadores con guiones. Ejemplo: si el usuario pide A1468 y el mejor resultado local no contiene A1468, se fuerza fallback web aunque haya resultados parecidos.

Como replicar el POC

Configuracion

  1. Crear llm-agent-project/.env desde .env.example.
  2. Agregar GROQ_API_KEY para el LLM.
  3. Agregar LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY y LANGFUSE_HOST para observabilidad.
  4. Agregar SERPAPI_API_KEY para fallback web real con Google Shopping.

Ejecucion

  1. cd llm-agent-project
  2. .venv\Scripts\python.exe -m pip install -r requirements.txt
  3. .venv\Scripts\python.exe scripts/ingest_catalog_to_chroma.py
  4. .venv\Scripts\python.exe -m streamlit run app/main.py
