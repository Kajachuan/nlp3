Procesamiento de Lenguaje Natural 3 - Trabajo práctico

Asistente de compra de componentes electrónicos

POC multiagente para responder consultas naturales con componente, precio y link de compra verificable.

Integrantes: Leandro Saraco y Kevin Cajachuán

1. Problema

Buscar componentes no es solo buscar texto

El usuario pregunta en lenguaje natural, pero espera una respuesta accionable: componente, precio y link de compra.

Desafíos

  • Consultas con SKU exacto, descripción técnica o errores de tipeo.
  • Necesidad de priorizar un catálogo propio antes de salir a la web.
  • Evitar respuestas sin evidencia o fuera del dominio del asistente.
2. Arquitectura general

Un flujo modular orquestado con LangGraph

Componentes principales

  • UI conversacional en Streamlit.
  • Planner LLM para intención y ruta.
  • Búsqueda local híbrida: Chroma + Keyword/BM25.
  • Fallback web con SerpAPI, Google Shopping o Tavily.
  • Agregador/verificador para respuesta final.
3. Pipeline

La consulta atraviesa seis decisiones

02

03

Busca por embeddings y por coincidencias léxicas.
04

Evalúa score, precio y términos específicos.
05
Web fallback
Consulta proveedores externos si la evidencia local no alcanza.
06

Agrega evidencia, responde y registra auditoría.

Preparación del RAG

  • La ingesta persiste una colección local en ChromaDB.
  • El sistema también puede autoindexar al iniciar la app si falta la colección.
5. Retrieval local

Búsqueda semántica + búsqueda léxica

ChromaDB

Recupera candidatos semánticamente cercanos y calcula un score vectorial normalizado.

Keyword/BM25

Refuerza SKU, part numbers, nombres exactos y términos técnicos que no conviene diluir.

Merge híbrido

Combina 65% semántico, 35% keyword y agrega bonus si un item aparece por ambas vías.

6. Compuerta de evidencia

No siempre conviene responder con lo local

Condiciones para usar fallback web

  • No hay suficientes resultados locales.
  • El mejor score queda debajo del umbral.
  • Faltan términos específicos como SKU o part number.
  • El usuario necesita precio y el item local no lo tiene.

Por qué importa

La compuerta evita que un resultado parecido pero incorrecto contamine la respuesta final. Es el puente entre retrieval y decisión de compra.

7. Web y proveedores

La web entra como respaldo, no como primera opción

WebSearchAgent

Consulta el adapter externo, normaliza resultados y conserva vendor, precio, score, snippet y URL.

Backends

Tavily, Google Shopping y Google Search mediante credenciales configurables.

8. Planner y verificador

Dos agentes que

interactuan entre si

Planner

  • Define ruta: catálogo local, fuera de dominio o info del agente.
  • Extrae términos de producto y prepara query web.
  • Evita llamar herramientas cuando no corresponde.

Agregador/verificador

  • Recibe evidencia local y externa estructurada.
  • Elige fuente confiable, precio y link.
  • Puede abstenerse si no hay soporte suficiente.

10. Seguridad y auditoría

        
Controles simples pero importantes

      

      

        

          
Validación

          
Bloquea consultas vacías, demasiado largas, secretos, scripts y payloads peligrosos.

        

        

          
Límites

          
Clampea parámetros como top_k y web_limit para controlar costo y latencia.

        

        

          
Audit log

          
Registra ruta elegida, fallback usado y métricas de cada respuesta en JSONL.
